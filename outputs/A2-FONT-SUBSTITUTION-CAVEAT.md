# 已知限制：教材 PDF 数学变量字形替换（人教版排版特征）

**发现于**：W3 A2 单本 PDF dry run（`RENJIAO-G8-MATH-S` / 人教版·数学八年级上册）
**性质**：PDF 字体层面的结构性限制，非 Knowledge Hub 管道 bug
**处置**：**接受为已知限制，按原计划批量入库**（2026-07-18 用户拍板）

## 现象

用 `fitz`（PyMuPDF）抽取文本时——无论走 `page.get_text("text")` 纯文本模式，
还是 `embed_chunks.py` 当前用的 span 逐段拼接——同一页产出完全一致的结果，
说明这**不是抽取方式的选择问题**。

抽样发现：
- **纯散文段落**（无内联数学符号）抽取完全干净、可读。
- **含内联数学变量的段落**：变量字母被替换成形状相似但语义无关的真实汉字，
  例如 `x`→`狓`、`a`→`犪`、`m`→`犿`。跨多章多页一致出现（第14/16/18章均有），
  说明这是该出版社（人教版）数学教材排版系统的**系统性字体映射约定**，不是
  个别页面的偶发损坏。
- **少数章节扉页**：出现一小段不可读的控制字符/私用区乱码（如
  `\x12\x19\x0f\x16`、`ʷǌ ቦᅼគፇ ڏ̄`），来自一个特定字体
  （`FZLTZHJW--GB1-0-GBK-EUC-`），推测是扉页装饰图形/字号被误判成文字层，
  该字体没有可用的 ToUnicode 映射——文本层完全无法恢复，只影响这些扉页，
  不影响正文。

## 根因

这是 PDF **字体层面**的问题：涉事字体把标准 Unicode 码位重新映射到自定义
字形（常见于方正等中文数学排版系统，为保证公式里拉丁字母与中文数字混排的
版式一致性）。字体本身"撒谎"字符含义，任何通用文本抽取库（fitz/pypdf/
pymupdf4llm 皆基于同一份 PDF 文本层）都无法在不引入 OCR 或该出版社专属字形
映射表的前提下恢复正确字符——换库不能解决，已用两种 fitz 抽取模式验证过
完全一致的输出。

## 影响

- **A-3 验收（"抽样 chunk 文本与原 PDF 页码一致"）严格意义上对含内联数学符号
  的 chunk 不完全成立**——变量字母不是其视觉/语义原意。
- **不影响**：纯散文 chunk（章节导言、应用题文字部分等）、页码/章节定位
  （`page_number`/`char_span` 仍精确对应原文字符位置，只是该位置的字符本身
  被字体替换了）、`pdf_id`/出处链路（学生可凭页码翻到原页，看到正确渲染的
  数学排版——这是纸面/图像层，不受文本层字体映射问题影响）。
- 检索（embedding 相似度）主要靠散文语境定位内容，多数概念性提问预期仍可用；
  精确复现公式原文的场景（如直接引用某道例题的完整符号表达式）会失真。

## 决定

用户拍板：**接受为已知限制，按计划批量入库全部 22 本**，不额外投入 OCR 或
字形映射表工作。理由：出处能溯源到具体文件+页+字符区间已达标（W3 spec 的
`pdf_id`/`char_span`/`textbook_meta` 要求不要求文本本身像素级正确），且
页码引用让学生可回原书核对真实公式。

如未来需要更高保真度（如 Book Engine 要直接引用/渲染教材例题原文），需重新
评估 OCR 管道或该出版社专属字形映射表——留后续 task，不在 W3 范围内。
